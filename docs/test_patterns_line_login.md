# LINEログイン機能 手動テストパターン

## テスト準備
- LINE Developersコンソールで対象チャネルの設定を確認
  - Channel ID / Channel Secret が `.env` の `LINE_CHANNEL_ID` / `LINE_CHANNEL_SECRET` と一致
  - コールバックURLに `http://localhost:3000/users/auth/line/callback` が登録済み(dev用)
  - メールアドレス取得申請が承認済み
- 開発用のLINEアカウントを2つ用意（A: メアド共有あり / B: メアド共有なしで挙動確認用）
- `docker compose up` でサーバー起動済み
- DB上に同じメアドの既存ユーザー(Google連携 or 通常)が存在するパターンを用意

---

## 1. 新規ユーザーのLINEログイン(メアド取得成功)

### 1-1. ログイン画面からの新規登録
- [ ] 未ログイン状態で `/users/sign_in` を開く
- [ ] 「LINEでログイン」ボタンが LINE緑(#06C755) + LINEロゴ + 白文字で表示されている
- [ ] ボタンをクリック → LINE認可画面に遷移する
- [ ] LINE側でログイン → 「メールアドレスを共有する」を許可して同意
- [ ] アプリにリダイレクトされ、ログイン状態でトップページに遷移する
- [ ] フラッシュメッセージ「LINEアカウントによる認証に成功しました」等が表示される

### 1-2. サインアップ画面からの新規登録
- [ ] 未ログイン状態で `/users/sign_up` を開く
- [ ] 「LINEでログイン」ボタンが表示されている(同上のスタイル)
- [ ] クリック → LINE認可 → ログイン完了

### 1-3. DB上のユーザーレコード確認
- [ ] `rails console` または `/admin/users` で新規ユーザーが作成されていること
- [ ] `provider = "line"` / `uid` がLINEのuserId(U〜)で保存されている
- [ ] `email` がLINE側のメアド
- [ ] `name` がLINE側の表示名(displayName)
- [ ] `encrypted_password` が空ではない(ランダム生成済み)
- [ ] `current_sign_in_at` が記録されている、`current_sign_in_ip` は **nil**(IP記録しない仕様)

## 2. 既存LINEユーザーの再ログイン

### 2-1. 同一アカウントでの再ログイン
- [ ] 1で作成したアカウントでログアウト
- [ ] 再度「LINEでログイン」ボタンをクリック → LINE認可画面 → 同意
- [ ] **新規ユーザーが作成されない**(同じuserIdのレコードが返る)
- [ ] `sign_in_count` がインクリメントされている
- [ ] `current_sign_in_at` が更新されている

### 2-2. ログイン状態の保持
- [ ] ログイン後に他ページ(/articles など)に遷移してもログイン状態が維持される
- [ ] ブラウザを閉じて再度開いても、`remember_me` 等で適切に復帰する

## 3. メアド未取得時の補完フロー

### 3-1. メアド共有を許可しなかった場合
- [ ] LINE認可画面で「メールアドレスを共有する」のチェックを外して同意
- [ ] アプリに戻ると `/users/auth/line/email_setup` へリダイレクトされる
- [ ] 「メールアドレスの登録」画面が表示される
- [ ] 「LINEからメールアドレスを取得できませんでした」の説明文が表示されている
- [ ] LINEアカウント名が表示されている(`LINEアカウント名: XXX`)

### 3-2. 補完画面の表示要素
- [ ] メールアドレス入力欄(autofocus)
- [ ] 「メールアドレスを登録してログイン」ボタン(LINE緑)
- [ ] 「キャンセル」リンク → クリックで `/users/sign_in` に戻る
- [ ] 利用規約 / プライバシーポリシーへのリンクが表示される

### 3-3. 補完成功
- [ ] 未使用のメアドを入力 → 「メールアドレスを登録してログイン」をクリック
- [ ] ログイン状態でトップページに遷移
- [ ] DBに新規Userレコードが作成されている(provider:line / uid / email:入力値 / name:LINE名)

### 3-4. キャンセル動作
- [ ] 補完画面で「キャンセル」をクリック → `/users/sign_in` にリダイレクト
- [ ] その後再度LINEログインを試みると、補完画面に再度遷移する(セッションが残っているため)

---

## 4. 補完画面のバリデーション

### 4-1. 既存メアドとの衝突
- [ ] 補完画面で **既に登録済みの別ユーザーのメアド** を入力 → 送信
- [ ] 補完画面に留まり、エラーメッセージ「メールアドレスが既に登録されているか、入力内容が無効です。」が表示される
- [ ] DBに重複ユーザーが作成されていない

### 4-2. メールアドレス形式不正
- [ ] `abc` のような不正なメアドを入力 → 送信
- [ ] エラーメッセージが表示されて補完画面に留まる
- [ ] 不正レコードが作成されていない

### 4-3. メアド空欄
- [ ] 空欄のままsubmit → ブラウザのHTML5バリデーション(required)でブロックされる

### 4-4. セッション切れ
- [ ] 補完画面を表示した状態で、別タブからアプリのログアウトやセッションクリア
- [ ] 補完画面でsubmit → `/users/sign_in` にリダイレクトされ「セッションが無効です。再度LINEログインからお試しください。」のalertが出る

## 5. 既存メアドとの自動紐付けは行わない(別アカウント扱い)

### 5-1. Google連携済みメアドと同じLINEメアド

- [ ] 同じメアド(`foo@example.com`)が登録されたLINEアカウントで「LINEでログイン」
- [ ] ログイン**失敗** → サインアップ画面にリダイレクト
- [ ] alert「LINEログインに失敗しました。同じメールアドレスのアカウントが既に登録されている可能性があります。」
- [ ] DBに**新規LINEユーザーが作成されていない**(別アカウントとして紐付けない仕様)

### 5-2. 通常Devise登録済みメアドと同じLINEメアド
- [ ] パスワード登録済みのDeviseユーザーが `foo@example.com` で存在
- [ ] 同じメアドのLINEアカウントでログイン → 同じく失敗・新規作成されない

## 9. エラー系

### 9-1. LINE側でキャンセル
- [ ] LINE認可画面で「キャンセル」を押す
- [ ] アプリ側のfailureアクションが呼ばれて `/users/sign_in` にリダイレクト
- [ ] alert「認証に失敗しました。もう一度お試しください。」が表示される

### 9-2. callback処理中のエラー
- [ ] LINEのid_token verify APIが異常応答を返した場合

  - `tail -f log/development.log` で `[OmniAuth] (line) verify API ...` のログが出ること

### 10-3. コールバックURL不一致時の挙動確認(任意)

- [ ] LINE認可画面後にエラーになることを確認(完全一致が必須なため)

## トラブルシュート

| 症状 | 想定原因 | 確認 |
| --- | --- | --- |
| `passthru` で 404になる | サーバ未再起動 / Strategyロード失敗 | `bin/dev` 再起動、`rails runner 'puts defined?(OmniAuth::Strategies::Line)'` |
| nonce検証失敗のログ | session欠落・改竄・cookie store容量超過 | session_storeサイズ、cookieサイズ確認 |
| メアドが取得できない | LINE Developersのメアド取得申請が未承認 / scopeに `email` がない | Strategy の scope 確認、申請ステータス確認 |
| `LINE Developers callback URL mismatch` | 登録URL ≠ 実際のcallback URL | LINE Developers / `LINE_CALLBACK_URL` ENVを完全一致させる |
